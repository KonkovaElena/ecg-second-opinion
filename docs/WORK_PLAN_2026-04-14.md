## Next Recommended Waves

### Wave 1 — Durable Inference Delivery

Goal: remove the highest operational risk in the current architecture.

Tasks:

1. introduce an inference dispatch abstraction instead of calling `setImmediate()` directly in `src/app.ts`
2. add a retry/backoff policy with bounded attempts
3. add a dead-letter state or replayable failure record
4. add tests that prove cases cannot remain silently stranded without an observable failure outcome

Acceptance criteria:

- inference dispatch is replaceable in tests
- retry policy is explicit and bounded
- failures remain observable and auditable

### Wave 2 — Real Persistence

Goal: remove restart data loss and make audit history durable.

Tasks:

1. add a SQLite-backed repository adapter behind `IEcgCaseRepository`
2. preserve audit trail and safety-flag resolution semantics
3. add integration tests for save/load/reconstitute flows

Acceptance criteria:

- restart-safe case persistence
- aggregate reconstitution proven by tests
- no state-machine regression from storage round trips

### Wave 3 — Better Interoperability

Goal: harden the report surface for external clinical systems.

Tasks:

1. add coded vocabularies to FHIR export
2. add `meta.profile` support
3. define stronger identifier-system rules
4. add at least one validator-backed FHIR test path

Acceptance criteria:

- export is still human-readable
- coded values are machine-usable
- validator-backed tests cover the main shape

### Wave 4 — Operational Observability

Goal: improve runtime accountability and operator debugging.

Tasks:

1. add structured request logging middleware
2. derive or harden the awaiting-review gauge
3. improve shutdown behavior for in-flight inference work

Acceptance criteria:

- request logs include method, path, status, latency, correlation id
- metrics do not drift negative or silently stale
- shutdown behavior is explicit and documented

## Recommended Execution Order

1. Wave 1 first: it addresses the highest-risk runtime hole.
2. Wave 2 second: it turns the workflow from memory-local to durable.
3. Wave 3 third: it improves external interoperability once runtime and storage are credible.
4. Wave 4 can run in parallel with Wave 2 or Wave 3 if needed.
