# ECG Second Opinion

[English](README.md) | Русский

**Система второго мнения по ЭКГ с обязательным врачебным контролем.**

`ECG Second Opinion` — это отдельный TypeScript API, который ведёт кейс ЭКГ от приёма записи через асинхронный inference и клинические safety checks до обязательного просмотра врачом, финализации и передачи результата.

> **Важно. Только для исследований и инженерной оценки.** Репозиторий не одобрен для клинического применения и не может использоваться для постановки диагноза без проверки квалифицированным специалистом.

Проект не позиционируется как «ИИ, который сам читает ЭКГ». Это рабочий контур вокруг модели, где человек остаётся обязательной точкой принятия решения.

## Что делает проект

- принимает 12-канальные записи ЭКГ и сопутствующие метаданные;
- ставит кейс в очередь на асинхронную обработку;
- сохраняет результат алгоритма как структурированный черновик, а не как окончательный диагноз;
- прогоняет результат через клиническую safety policy из 8 правил;
- требует явного разбора и решения врача перед финализацией;
- экспортирует результат в FHIR R4 DiagnosticReport и во внутренний структурированный ECG report;
- ведёт журнал состояния и аудита по кейсу.

## Что подтверждено текущей реализацией

- 6-состояний доменного агрегата кейса;
- трёхконтурная аутентификация: API key, JWT HS256 для reviewer и internal bearer token;
- строгая Zod-валидация на всех маршрутах;
- safety policy с 8 клиническими правилами;
- FHIR R4 DiagnosticReport export;
- Prometheus-метрики, health probes и корреляционные идентификаторы;
- контейнерная упаковка в non-root Docker image.

## Что проект не заявляет

- автономную диагностику;
- клинический триаж без врача;
- завершённый production-grade inference worker;
- регуляторное одобрение или клиническую готовность;
- платформу для обучения модели внутри этого репозитория.

## Безопасность

- операторские маршруты защищаются `x-api-key`;
- врачебные действия используют JWT HS256 с проверкой `alg`, `iss`, `aud`, `exp`, `nbf`, `sub` и `role`;
- внутренние callback-маршруты закрываются отдельным bearer token;
- сравнение секретов делается через `crypto.timingSafeEqual`;
- входные данные валидируются строгими Zod-схемами;
- включены rate limiting, Helmet и non-root container baseline;
- в логи не попадают реальные идентификаторы пациентов.

## Пример рабочего маршрута

1. Оператор отправляет кейс через `POST /api/v1/cases`.
2. Сервис переводит кейс в `InferencePending`.
3. Внутренний inference-контур формирует черновик и safety flags.
4. Врач разбирает кейс и отправляет решение через reviewer JWT.
5. После этого кейс может быть финализирован и экспортирован.

## Технологический стек

- Node.js 22+, TypeScript 5.8 и Express 4;
- Zod для контрактной валидации;
- `prom-client` для метрик;
- Jest, `ts-jest` и `supertest` для тестов;
- встроенный `node:crypto.randomUUID()` для идентификаторов;
- Docker multi-stage с non-root пользователем.
